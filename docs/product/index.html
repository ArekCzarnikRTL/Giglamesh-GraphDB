GraphMesh

KI-Knowledge-Graph-Plattform — Automatische Wissensextraktion aus Dokumenten, semantische Suche und KI-gestuetzte Abfragen ueber verknuepftes Wissen.

GraphMesh in 60 Sekunden — Von Dokumenten zu Wissen

01 · Ueberblick

Was ist GraphMesh?

GraphMesh ist eine KI-Knowledge-Graph-Plattform zur automatischen Wissensextraktion aus Dokumenten. Die Plattform analysiert Dokumente mithilfe von KI-Sprachmodellen, extrahiert strukturiertes Wissen als Wissensgraph (Subjekt-Praedikat-Objekt-Tripel) und ermoeglicht semantische Suche sowie intelligente Abfragen ueber den gesamten Wissensbestand.

Der zentrale Gedanke: Unstrukturiertes Wissen in Dokumenten wird automatisch in einen durchsuchbaren, verknuepften Wissensgraphen ueberfuehrt — ohne manuelle Verschlagwortung oder Modellierung.

Wissen in Dokumenten ist unzugaenglich

Informationen stecken in PDFs, Reports und Texten, die nur durch manuelles Lesen erschlossen werden koennen. GraphMesh extrahiert Wissen automatisch und macht es sofort durchsuchbar.

Manuelle Wissensextraktion skaliert nicht

Das haendische Erfassen von Zusammenhaengen aus Hunderten von Dokumenten ist zeitaufwaendig und fehleranfaellig. KI-gestuetzte Extraktion arbeitet automatisch und konsistent.

Stichwortsuche reicht nicht

Klassische Volltextsuche findet keine semantischen Zusammenhaenge und versteht keine natuerlichsprachlichen Fragen. GraphMesh bietet Graph RAG, Document RAG und NLP-Abfragen.

Keine Nachvollziehbarkeit

Bei klassischen KI-Antworten ist unklar, woher die Information stammt. GraphMesh liefert Quellenangaben, Relevanz-Scores und Erklaerungen zu jeder Antwort.

02 · Dokumentenverwaltung

Upload, Verarbeitung und Organisation

GraphMesh organisiert Dokumente in Collections (thematische Sammlungen) und verarbeitet hochgeladene Dateien automatisch: Texterkennung, Aufteilung in semantische Abschnitte und Vorbereitung fuer die Wissensextraktion.

Collection-Verwaltung im Admin-Bereich
Collections

Ohne Struktur landen alle Dokumente in einem grossen Topf — Suchergebnisse werden ungenau, Zugriffssteuerung ist unmoeglich.

Collections buendeln zusammengehoerige Dokumente unter einem Namen mit optionaler Beschreibung und Tags. Jede Abfrage (Suche, RAG, Graph) arbeitet auf genau einer Collection.

# Alle Collections auflisten (optional nach Tags filtern)
query {
  collections(tags: ["forschung"]) {
    id
    name
    description
    tags
  }
}

# Neue Collection anlegen
mutation {
  createCollection(input: {
    name: "Forschungsberichte"
    description: "..."
    tags: ["forschung"]
  }) {
    id
    name
  }
}
Dokumenten-Upload

Dokumente muessen einzeln und manuell verarbeitet werden — es gibt keinen einheitlichen Einstiegspunkt fuer verschiedene Formate.

GraphMesh nimmt Dokumente per GraphQL-API, Frontend oder CLI entgegen. Nach dem Upload startet die Verarbeitungspipeline automatisch: PDF-Decode, Chunking, Embedding und Extraktion.

# Dokument hochladen
mutation {
  uploadDocument(input: {
    collectionId: "col-123"
    title: "Jahresbericht"
    mimeType: "application/pdf"
    content: "..."
  }) {
    id
    title
    state
  }
}

# Dokumente auflisten und filtern
query {
  documents(
    collectionId: "col-123"
    filter: { type: SOURCE state: EXTRACTED }
    page: 0
    pageSize: 20
  ) {
    items { id title state }
    totalCount
  }
}
PDF-Decode

PDF-Dateien enthalten Texte in komplexen Layouts — Spalten, Tabellen, Kopf-/Fusszeilen erschweren die maschinelle Verarbeitung.

GraphMesh extrahiert Text seitenweise aus PDFs und wandelt ihn in strukturierte Textdokumente um. Die Pipeline erkennt Seitengrenzen und erzeugt ein Dokument pro Seite als Grundlage fuer das Chunking.

Semantisches Chunking

Ganze Dokumente sind zu lang fuer LLM-Verarbeitung. Willkuerliches Aufteilen zerstoert Zusammenhaenge.

GraphMesh teilt Dokumente in semantisch sinnvolle Abschnitte (Chunks) auf. Die Chunk-Groesse und Ueberlappung sind konfigurierbar. Jeder Chunk wird mit Vektor-Embeddings versehen und ist einzeln durchsuchbar.

03 · Wissensextraktion

Automatische Extraktion von Wissen mittels KI

Nach dem Upload verarbeitet GraphMesh Dokumente automatisch: Der Text wird in Abschnitte aufgeteilt, und aus jedem Abschnitt extrahiert das System strukturiertes Wissen. Die Extraktion laeuft im Hintergrund, sobald ein Dokument hochgeladen wurde.

Beziehungsextraktion

In Freitexten stecken Fakten und Zusammenhaenge, die fuer Suche und Analyse nur nutzbar sind, wenn sie strukturiert vorliegen.

GraphMesh erkennt automatisch Beziehungen in Textabschnitten und speichert sie als Subjekt-Praedikat-Objekt-Tripel (z.B. "Firma X" — "hat Hauptsitz in" — "Berlin"). Diese Tripel bilden den Knowledge Graph.

Ontologie-gestuetzte Extraktion

Ohne Vorgaben nutzt die KI inkonsistente Bezeichnungen — "Firmensitz", "Hauptsitz" und "Standort" meinen dasselbe, werden aber als verschiedene Praedikate gespeichert.

Durch den Import einer Ontologie erhaelt die Extraktion einen Leitfaden. Die KI bevorzugt Begriffe aus der Ontologie und erzeugt konsistentere, besser verknuepfte Tripel.

Strukturierte Extraktion

Neben freien Beziehungen werden haeufig konkrete Datentypen benoetigt — Zahlen, Datumsangaben, Klassifikationen.

GraphMesh unterstuetzt Datatype Properties, die Entitaeten mit typisierten Werten verknuepfen. So werden neben Beziehungen auch konkrete Fakten wie Mitarbeiterzahl oder Gruendungsdatum erfasst.

Agenten-basierte Extraktion

Einfache Prompt-basierte Extraktion erreicht bei komplexen Dokumenten ihre Grenzen — Kontext geht verloren, Ambiguitaeten werden nicht aufgeloest.

Der Extraktions-Agent arbeitet in einer Denk-Handlungs-Schleife: Er analysiert den Text, konsultiert die Ontologie, loest Mehrdeutigkeiten auf und prueft die Konsistenz extrahierter Tripel.

Themenextraktion

Einzelne Tripel zeigen Detailwissen, aber nicht das uebergeordnete Thema eines Dokuments oder Abschnitts.

GraphMesh erkennt automatisch Themen und Schlagwoerter aus Dokumenten-Chunks und ordnet sie in die SKOS-Taxonomie ein. So lassen sich Dokumente nach Themengebieten filtern und navigieren.

04 · Suche & Abfragen

Semantische Suche und RAG-Abfragen

GraphMesh bietet verschiedene Wege, Wissen aus Ihren Dokumenten und dem Knowledge Graph abzufragen — von gezielten Graph-Abfragen ueber semantische Dokumentensuche bis hin zu automatisch optimierten Freitext-Fragen.

Beispiel einer natuerlichsprachlichen Abfrage mit Ergebnis und Quellenangaben
Graph RAG

Traditionelle Suchsysteme finden nur Dokumente mit bestimmten Begriffen. Zusammenhaenge zwischen Konzepten bleiben verborgen.

Graph RAG durchsucht den Knowledge Graph nach relevanten Wissens-Beziehungen und generiert eine Antwort, die auf konkreten Fakten (Tripeln) basiert. Jede Antwort nennt die verwendeten Beziehungen mit Relevanz-Score.

query {
  graphRag(input: {
    question: "Welche Technologien nutzt Projekt Alpha?"
    collectionId: "..."
    maxEdges: 150
    maxDepth: 2
  }) {
    answer
    selectedEdges {
      subject
      predicate
      objectValue
      relevanceScore
    }
  }
}
Document RAG

Bei grossen Dokumentenbestaenden ist es schwierig, die relevante Textstelle zu finden. Volltextsuche versagt bei umformulierten Fragen oder Synonymen.

Document RAG sucht semantisch in Dokumenten-Chunks und generiert eine quellenbasierte Antwort. Jede Antwort verweist auf die konkreten Dokumente, Seiten und Textausschnitte.

query {
  documentRag(input: {
    question: "..."
    collectionId: "..."
    topK: 10
  }) {
    answer
    sources {
      documentTitle
      pageNumber
      score
      snippet
    }
  }
}
NLP Query

Nutzer muessen wissen, ob sie Graph RAG oder Document RAG nutzen sollen — die optimale Suchstrategie haengt von der Frage ab.

NLP Query analysiert die Frage automatisch, erkennt die Absicht (Intent) und waehlt die passende Suchstrategie. Das Ergebnis enthaelt die erkannte Absicht mit Konfidenz-Score.

query {
  nlpQuery(input: {
    question: "..."
    collectionId: "..."
  }) {
    answer
    detectedIntent {
      intent
      confidence
    }
  }
}
Explainability

KI-generierte Antworten sind oft intransparent — der Nutzer weiss nicht, auf welcher Grundlage die Antwort basiert.

Jede GraphMesh-Antwort enthaelt Quellenangaben: Bei Graph RAG die verwendeten Tripel mit Relevanz-Score und Begruendung, bei Document RAG die Quelldokumente mit Seitenzahl und Textausschnitt.

05 · Wissensmodellierung

Ontologien, Taxonomien und RDF-Import

GraphMesh ermoeglicht die Strukturierung von Wissen durch Ontologien, kontrollierte Vokabulare (Taxonomien) und den Import bestehender Wissensdaten. Diese Werkzeuge definieren, wie Wissen organisiert, klassifiziert und miteinander verknuepft wird.

Ontology System

Ohne definierte Struktur extrahiert ein LLM Wissen in beliebigen Formaten — "Mitarbeiter arbeitet-bei Firma" und "Person ist-angestellt-bei Unternehmen" beschreiben dasselbe, werden aber als unterschiedlich behandelt.

Ontologien definieren die erlaubten Klassen (z.B. Person, Organisation, Projekt) und Beziehungstypen (z.B. arbeitetBei, gehoertZu). GraphMesh nutzt diese Definitionen, um Wissen konsistent zu extrahieren und zu speichern.

SKOS-Taxonomien

Freitext-Schlagwoerter fuehren zu inkonsistenter Verschlagwortung — "KI", "Kuenstliche Intelligenz" und "AI" meinen dasselbe.

SKOS (Simple Knowledge Organization System) bietet hierarchische Begriffssysteme mit Synonymen, Ober- und Unterbegriffen. GraphMesh nutzt SKOS-Taxonomien zur einheitlichen Klassifikation von Wissen.

RDF-Import

Bestehendes strukturiertes Wissen (aus Datenbanken, Linked-Data-Quellen) muss manuell neu erfasst werden.

GraphMesh importiert RDF-Daten in Standardformaten (Turtle, RDF/XML, N-Triples). Bestehende Wissensgraphen koennen direkt in das System uebernommen und mit extrahiertem Wissen kombiniert werden.

06 · Graph-Visualisierung

Interaktive Exploration des Wissensgraphen

GraphMesh stellt extrahiertes Wissen als interaktiven Wissensgraphen dar. Entitaeten und ihre Beziehungen lassen sich visuell erkunden, filtern und schrittweise erweitern.

Graph Explorer mit Sigma.js-Visualisierung: Entitaeten und Beziehungen als interaktives Netzwerk
Graph Explorer

Tabellarische Listen von Wissenseintraegen zeigen keine Zusammenhaenge. Nutzer koennen nicht erkennen, wie Entitaeten miteinander verbunden sind.

Der Graph Explorer auf der /graph-Seite stellt Entitaeten als Knoten und Beziehungen als gerichtete Kanten dar. Das Layout ordnet sich automatisch per Force-Directed-Algorithmus an und passt sich dynamisch an.

07 · Agent & Streaming

KI-Agenten mit Werkzeugen und Echtzeit-Streaming

GraphMesh bietet KI-Agenten, die komplexe Fragen ueber den Wissensgraphen beantworten. Die Agenten nutzen Werkzeuge in einer Denk-Handlungs-Schleife (ReAct) und liefern Antworten in Echtzeit per Streaming.

Agent-System

Einfache Suchanfragen genuegen nicht, wenn die Antwort aus mehreren Wissensquellen zusammengesetzt werden muss.

GraphMesh-Agenten arbeiten in einer ReAct-Schleife: Sie analysieren die Frage, waehlen passende Werkzeuge, fuehren Aktionen aus, beobachten die Ergebnisse und wiederholen bis die Frage beantwortet ist.

mutation {
  askAgent(input: {
    question: "..."
    collectionId: "..."
    maxIterations: 10
  }) {
    sessionId
    answer
    durationMs
  }
}
Tool-Gruppen

Nicht jede Frage erfordert alle verfuegbaren Werkzeuge. Irrelevante Tools erhoehen die Antwortzeit und koennen den Agenten ablenken.

Werkzeuge sind in logische Gruppen organisiert (z.B. "Graph-Suche", "Dokumenten-Suche", "Vektor-Suche"). Bei jeder Anfrage kann festgelegt werden, welche Tool-Gruppen der Agent nutzen darf.

Echtzeit-Streaming

Agent-Anfragen koennen mehrere Sekunden dauern. Ohne Streaming sieht der Nutzer nur einen Ladebalken und erhaelt die Antwort erst am Ende.

GraphMesh streamt den gesamten Agenten-Prozess in Echtzeit. Jedes Token traegt einen Typ, der den aktuellen Schritt des Agenten kennzeichnet. Im Frontend wird eine farbcodierte Timeline dargestellt.

subscription {
  agentStream(input: {
    question: "..."
    collectionId: "..."
  }) {
    content
    type
    endOfMessage
    endOfStream
  }
}

Token-Typen

Typ Bedeutung Farbe
TEXT Allgemeiner Text Grau
THOUGHT Denkschritt des Agenten Blau
ACTION Werkzeug-Aufruf Gelb/Orange
OBSERVATION Ergebnis eines Werkzeugs Violett
ANSWER Finale Antwort Gruen
ERROR Fehlermeldung Rot
08 · Context Cores

Versionierte Wissens-Bundles: Export, Import, Tagging

Context Cores sind portable, versionierte Wissens-Bundles im ZIP-Format. Sie enthalten den vollstaendigen Stand einer Collection — Wissensgraph-Tripel, Vektor-Embeddings, Ontologie und Retrieval-Konfigurationen — und funktionieren wie Build-Artefakte fuer Wissen.

Context Core bauen (Export)

Der aktuelle Wissensstand einer Collection laesst sich nicht als geschlossenes Artefakt sichern, einfrieren oder weitergeben.

Die Build-Funktion exportiert den vollstaendigen Inhalt einer Collection als versioniertes ZIP-Bundle: N-Quads-Tripel, Ontologie (Turtle), Embeddings, Retrieval-Policies und ein Manifest mit SHA-256-Pruefsumme. Im Frontend unter /cores → Core bauen.

mutation {
  buildContextCore(
    coreId: "pharma-base"
    version: "1.0.0"
    sourceCollection: "meine-collection-id"
    description: "Pharma-Wissensbasis Q1 2026"
    tags: ["stage"]
    ontologyKey: "pharma-ontologie"
  ) {
    coreId version checksum
    stats { quadCount entityCount chunkEmbeddingCount }
  }
}
Context Core importieren

Extrahiertes Wissen aus einer Umgebung soll in eine andere Collection uebernommen werden, ohne die Dokumente erneut zu verarbeiten.

Die Import-Funktion laedt ein bestehendes Bundle in eine Ziel-Collection. Drei Konfliktstrategien stehen zur Verfuegung: FAIL (Abbruch wenn nicht leer), MERGE (Daten hinzufuegen) und REPLACE (vorhandene Daten ersetzen). Optional mit Namespace-Rewrite fuer URI-Praefixe.

mutation {
  importContextCore(
    coreId: "pharma-base"
    version: "1.0.0"
    targetCollection: "ziel-collection-id"
    strategy: MERGE
  ) {
    coreId version quadsImported embeddingsImported
  }
}
Versionierung & Tagging

Es gibt keinen Mechanismus um verschiedene Staende eines Wissensbestands zu unterscheiden oder als “produktionsreif” zu kennzeichnen.

Jeder Core traegt eine eindeutige Kombination aus Core-ID und Version. Zusaetzlich koennen Tags wie stage, prod oder latest vergeben werden. Ueber Tags laesst sich gezielt die aktuell gueltige Version abfragen — ideal fuer Promotion-Workflows.

# Tag setzen
mutation {
  tagContextCore(
    coreId: "pharma-base"
    version: "1.0.0"
    tag: "prod"
  ) { coreId version tags }
}

# Core per Tag abrufen
query {
  contextCoreByTag(
    coreId: "pharma-base"
    tag: "prod"
  ) { coreId version stats { quadCount } }
}
09 · Administration

Konfiguration, Mandanten und CLI

GraphMesh verwendet ein typisiertes Konfigurationssystem mit versionierten Eintraegen. Die Plattform ist mandantenfaehig und bietet neben der Web-Oberflaeche auch ein Kommandozeilen-Tool fuer die Verwaltung.

Konfigurationsverwaltung

Anwendungseinstellungen sind ueber Konfigurationsdateien verstreut und erfordern bei jeder Aenderung einen Neustart.

GraphMesh bietet ein zentrales Konfigurationssystem mit typisierten Eintraegen (LLM, Extraction, Embedding, Pipeline). Aenderungen werden versioniert und sind sofort wirksam — per Frontend (/admin/config), CLI oder GraphQL.

Multi-Tenancy

Verschiedene Teams oder Abteilungen muessen ihre Daten voneinander getrennt verwalten koennen.

GraphMesh unterstuetzt Mandantenfaehigkeit: Jeder Mandant hat eigene Collections, Dokumente und Wissensgraphen. Die Trennung erfolgt auf Datenebene — kein Mandant sieht die Daten eines anderen.

CLI-Referenz

Nicht alle Verwaltungsaufgaben lassen sich effizient ueber die Web-Oberflaeche erledigen — Batch-Operationen und Automatisierung erfordern ein Kommandozeilen-Tool.

Das graphmesh-CLI bietet Befehle fuer alle Kernfunktionen: Collection-Verwaltung, Dokumenten-Upload, Abfragen, Konfiguration und Ontologie-Import. Ideal fuer Scripting und CI/CD-Pipelines.

# Collections verwalten
graphmesh collection list
graphmesh collection create "Forschungsberichte" --tag forschung

# Dokument hochladen
graphmesh document upload report.pdf --collection col-123

# Abfragen
graphmesh query graphrag "Welche Risiken bestehen?" \
  --collection meine-sammlung

# Konfiguration
graphmesh config list --type llm
graphmesh config set model-name "gpt-4o" --type llm
10 · Fehlerbehebung

Troubleshooting

Haeufige Probleme bei der Nutzung von GraphMesh und deren Loesungen.

LLM-Verbindung

API-Key nicht konfiguriert: Extraktion schlaegt fehl, Fehlermeldung zur LLM-Verbindung im Log.

Loesung: Umgebungsvariable OPENAI_API_KEY bzw. ANTHROPIC_API_KEY setzen. In application.yml pruefen, ob der Provider aktiviert ist. Anwendung neu starten.

Provider antwortet nicht: Extraktion haengt oder bricht mit Timeout ab.

Loesung: Internetverbindung und Provider-Status pruefen. Modellname validieren (Standard: gpt-4o-mini). Bei Ollama: Server unter http://localhost:11434 erreichbar?

Dokument-Upload

PDF-Dekodierung fehlgeschlagen: Dokument bleibt im Status “Fehler”.

Loesung: PDF in Standard-Viewer pruefen. Kein Passwortschutz? Gescannte Dokumente benoetigen OCR-Vorverarbeitung.

Dokument nicht gefunden: “Document not found” nach Upload.

Loesung: Verarbeitung laeuft asynchron — wenige Sekunden warten. Status im Admin-Dashboard pruefen.

Abfrage-Probleme

Keine Ergebnisse: NLP-Abfrage liefert nichts oder irrelevante Treffer.

Loesung: Dokumente vorhanden und verarbeitet? Embeddings erzeugt? Abfrage praezisieren. Embedding-Modell pruefen (EMBEDDING_MODEL).

Timeout: Abfragen brechen bei grossen Collections ab.

Loesung: limit-Parameter verwenden. Qdrant-Service pruefen. Kleinere, praezisere Fragen formulieren.

Infrastruktur

Services nicht erreichbar: Cassandra, Qdrant, Kafka oder MinIO sind nicht gestartet.

Loesung: docker compose up -d startet alle Dienste. Einzeln pruefen:

  • Cassandra: Port 9042
  • Qdrant: curl http://localhost:6333/healthz
  • Kafka: Port 9092
  • MinIO: curl http://localhost:9000/minio/health/live
Frontend

Keine Daten in der UI: Netzwerkfehler oder leere Seiten.

Loesung: Backend unter http://localhost:8083/graphiql erreichbar? Falls anderer Port: NEXT_PUBLIC_GRAPHQL_URL setzen.

Streaming bricht ab: WebSocket-Verbindung schlaegt fehl.

Loesung: WebSocket-Endpunkt ws://localhost:8083/graphql pruefen. Bei Reverse-Proxy: WebSocket-Upgrade weiterleiten.

Allgemeine Tipps

Logs: logs/graphmesh.log (Level: DEBUG)
Alle Services starten: docker compose up -d
GraphiQL: http://localhost:8083/graphiql fuer interaktive API-Tests
Konfiguration: Alle Werte ueber Umgebungsvariablen setzbar (Defaults in application.yml)
